# Руководство по запуску и тестированию для QA-инженеров (QA Guide)

Это руководство предназначено для настройки окружения, запуска и ручного/автоматического тестирования сервера фискальных касс `superkassa-server`.

---

## 1. Подготовка окружения и запуск сервера

Сервер может работать в двух режимах:
* **DESKTOP** (локальный режим): использует базу данных SQLite и хранит настройки в локальном файле `core-settings.json`.
* **SERVER** (серверный режим): использует внешнюю базу данных (PostgreSQL / MySQL) и хранит все конфигурации в СУБД.

### Вариант А. Быстрый запуск в Docker (режим DESKTOP)
Для локального тестирования достаточно запустить контейнер, который использует SQLite и мок-ОФД:

1. Соберите исполняемый JAR-файл приложения:
   ```bash
   ./gradlew :server:bootJar
   ```
2. Соберите Docker-образ:
   ```bash
   docker build -t superkassa-server .
   ```
3. Запустите контейнер (с пробросом портов и монтированием тома для базы данных):
   ```bash
   docker run -d \
     -p 8080:8080 \
     -v superkassa-data:/app/data \
     --name superkassa \
     superkassa-server
   ```

*После этого сервер будет доступен по адресу `http://localhost:8080`.*

## 2. Доступ к OpenAPI и Swagger UI

Для интерактивного вызова методов API перейдите в браузере по ссылке:
👉 **[http://localhost:8080/swagger-ui/index.html](http://localhost:8080/swagger-ui/index.html)**

### Как авторизоваться в Swagger:
1. Нажмите кнопку **Authorize** в правом верхнем углу.
2. В появившемся окне введите ПИН-код администратора (по умолчанию `8888` при инициализации) или кассира в поле **Value**.
3. Нажмите **Authorize** и закройте окно. Все последующие запросы будут автоматически отправлять этот ПИН-код в заголовке `Authorization`.

## 3. Пошаговый сценарий тестирования (Happy Path)

Ниже приведен базовый цикл работы кассового аппарата.

### Шаг 1. Инициализация новой кассы (регистрация в системе)
Отправьте запрос для создания кассы на основе данных из ОФД (имитируется через Mock):
* **Метод**: `POST /kkm/init`
* **Тело запроса**:
  ```json
  {
    "ofdSystemId": "kkm-test-qa-1",
    "accessToken": "mock-ofd-token-12345",
    "adminPin": "8888"
  }
  ```
* **Ожидаемый ответ (200 OK)**: JSON с реквизитами созданной кассы (БИН, Название организации, Заводской номер и т.д.). Скопируйте возвращенный `id` кассы (UUID) — он понадобится для всех следующих запросов.

### Шаг 2. Открытие смены
* **Метод**: `POST /kkm/{kkmId}/shift/open`
* **Авторизация**: Передайте ПИН-код администратора `8888` в заголовке.
* **Ожидаемый ответ (200 OK)**: Возвращает объект `ShiftInfo` со статусом `OPEN` и временем открытия.

### Шаг 3. Создание пользователя-кассира (оператора)
Создадим обычного кассира для пробития чеков (чтобы не работать под администратором):
* **Метод**: `POST /kkm/{kkmId}/users`
* **Тело запроса**:
  ```json
  {
    "username": "Кассир Мария",
    "role": "CASHIER",
    "pin": "1111"
  }
  ```
* **Ожидаемый ответ (200 OK)**: Кассир создан. Теперь для фискальных операций можно использовать авторизацию с ПИН-кодом `1111`.

### Шаг 4. Продажа (регистрация чека)
Авторизуйтесь в Swagger под ПИН-кодом кассира (`1111`) и отправьте запрос продажи:
* **Метод**: `POST /kkm/{kkmId}/receipt/sell`
* **Тело запроса**:
  ```json
  {
    "items": [
      {
        "name": "Яблоки весовые",
        "price": 450,
        "quantity": 1.5,
        "vatGroup": "VAT_12",
        "measureUnitCode": "166"
      },
      {
        "name": "Пакет майка",
        "price": 20,
        "quantity": 1,
        "vatGroup": "NO_VAT",
        "measureUnitCode": "796"
      }
    ],
    "payments": [
      {
        "type": "CASH",
        "sum": 700
      }
    ]
  }
  ```
* **Ожидаемый ответ (200 OK)**: Чек успешно пробит. В ответе возвращается `documentId` (ИД фискального документа), `fiscalNumber` (номер чека) и фискальный признак.

### Шаг 5. Просмотр и печать чека
Вы можете проверить визуальное отображение чека на двух языках (казахском и русском):
* **HTML-версия**: Откройте в браузере: `http://localhost:8080/kkm/{kkmId}/documents/{documentId}/print.html`
* **PDF-версия**: Скачайте и проверьте PDF: `http://localhost:8080/kkm/{kkmId}/documents/{documentId}/print.pdf`

### Шаг 6. Закрытие смены (Z-отчет)
В конце рабочего дня смена должна быть закрыта:
* **Метод**: `POST /kkm/{kkmId}/shift/close`
* **Авторизация**: ПИН-код администратора `8888`.
* **Ожидаемый ответ (200 OK)**: Возвращается `ReportResult` с данными Z-отчета (суммы продаж по типам оплат, налогам и т.д.). Смена переходит в статус `CLOSED`.

## 4. Тестирование офлайн-режима и очереди доставки

Касса поддерживает автономный режим работы при сбоях связи с ОФД.

### Как проверить работу в офлайне:
1. Запустите контейнер с отключенным доступом к сети или укажите несуществующий адрес ОФД в настройках.
2. Пробейте чек продажи (`POST /kkm/{kkmId}/receipt/sell`).
3. **Ожидаемое поведение**: Чек будет пробит локально с автономным фискальным признаком (АФП), а ответ вернет статус доставки `OFFLINE_QUEUED`.
4. Проверьте список недоставленных документов в очереди:
   * **Метод**: `GET /kkm/{kkmId}/queue`
   * Документ должен находиться в списке со статусом ожидания отправки.
5. Восстановите сеть и принудительно запустите отправку накопившейся очереди:
   * **Метод**: `POST /kkm/{kkmId}/queue/retry-failed`
   * Очередь должна быть выгружена в ОФД, а статус документов сменится на `SENT`.

## 5. Просмотр логов и базы данных

### Логи контейнера:
Для просмотра логов в реальном времени используйте команду:
```bash
docker logs -f superkassa
```

### Просмотр локальной БД (SQLite):
База данных SQLite сохраняется на примонтированном томе. Чтобы скопировать её к себе на компьютер для анализа через любой SQLite-браузер (например, DBeaver):
```bash
docker cp superkassa:/app/data/superkassa.db ./superkassa_test.db
```
После этого вы можете открыть файл `superkassa_test.db` и посмотреть таблицы `cashboxes`, `kkm_users`, `fiscal_documents` и другие.
